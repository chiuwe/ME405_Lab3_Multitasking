27 /** \mainpage Inter-task communication tests
28  * This program implements a set of tests to verify that the ME405 library's
29  * inter-task communication works. It's also a usable example of how the queues and
30  * shared data items are used. In addition, it helps to show why non thread safe data
31  * transfer is a bad idea; there is a global variable used to pass data between two
32  * tasks, and as the program runs, the global variable sometimes gets corrupted.
33  *
34  * The task diagram for this program is shown below.
35  * \dot
36  * digraph task_diagram {
37  * node [ shape = box, fontname = FreeSans, fontsize = 10 ];
38  * task_multi1 [ label = "MultiTask 1\nPri: 1\n10-137 ms" URL = "\ref task_multi" ];
39  * task_multi2 [ label = "MultiTask 2\nPri: 1\n10-137 ms" URL = "\ref task_multi" ];
40  * task_multiN [ label = "MultiTask N\nPri: 1\n10-137 ms" URL = "\ref task_multi" ];
41  * task_sink [ label = "Data Sink\nPri: 2\n0-20 ms" URL = "\ref task_sink" ];
42  * task_source [ label = "Data Source\nPri: 2\n1 ms" URL = "\ref task_source" ];
43  * task_user [ label = "User Interface\nPri: 0\n1 ms" URL = "\ref task_user" ];
44  * edge [ arrowhead = normal, style = solid, fontname = FreeSans, fontsize = 10 ];
45  * task_source -> task_sink [ label = "queue_1\nuint32_t" ];
46  * task_source -> task_sink [ label = "share_1\nuint32_t", style = dashed ];
47  * task_source -> task_sink [ label = "glob_of_probs\nuint32_t", style = dotted ];
48  * task_multi1 -> task_user [ label = "print_ser_queue\nchar" ];
49  * task_multi2 -> task_user [ label = "print_ser_queue\nchar" ];
50  * task_multiN -> task_user [ label = "print_ser_queue\nchar" ];
51  * task_sink -> task_user [ label = "print_ser_queue\nchar" ];
52  * }
53  * \enddot
54  * There are a couple of unusual features on this task diagram. The number of
55  * "multi-tasks" can be varied by changing \c N_MULTI_TASKS in \c main(). These
56  * multi-tasks are a bunch of tasks, all created from the same class, that are used
57  * to put some stress on the RTOS by running lots of tasks at the same time. The
58  * multi-task's timing includes a delay of random duration; this timing scheme causes
59  * tasks to pre-empt each other with random timing. The data sink task normally runs
60  * very quickly, looking for changed data; if it finds data, it delays by 20 ms to
61  * give other tasks some time to run. The data sink therefore takes up a lot of CPU
62  * time, and one can see that the user interface runs slowly because of this.
63  *
64  * When this program is running, one can press the "v" key in a terminal window to
65  * cause the program to display a task status report. A typical report is shown
66  * below.
67  * \code
68  * Task Stack
69  * Name Pri. State Free/Total Runs
70  * ---- ---- ----- ---------- ----
71  * Multi3 1 0 63/120 0 153 runs
72  * Multi2 1 0 63/120 0 161 runs
73  * Multi1 1 0 63/120 0 157 runs
74  * Multi0 1 0 63/120 0 152 runs
75  * UserInt 1 0 66/240 0 4217 runs
76  * Sink 2 0 92/160 0 1136446 runs, errors in queue: 0, shared_data: 0, global data: 265
77  * Source 2 0 145/220 0 10446 runs
78  * IDLE 0 - 50/100
79  * Heap: 4000/5944, OCR5A=1999
80  * \endcode
81  * This report shows tasks and their current priorities and states. Also, for the
82  * data sink task, it shows a report of how accurately the data from the data source
83  * task has been received. One should see that there have been no errors in sending
84  * data through the queue or shared data item, but the use of a global variable has
85  * caused errors due to task switching while the data was being written or read.
86  */
